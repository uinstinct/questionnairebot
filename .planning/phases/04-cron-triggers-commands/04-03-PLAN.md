Render `/status` and `/list` (deterministic plain-text tables), expose SendPicker on the
bot, route `/pull`,`/status`,`/list` and inline callback queries through the dispatcher,
and replace the stub cron callback in main.go with `commands.CronBus.Fire`.
</objective>

<must_haves>
  <truths>
    <truth>/status fields per row: Name, Last (RFC3339 in q.Location or "Never"), Next (RFC3339 in q.Location), State (`✅ Done` / `🔄 In Progress` / `⏳ Pending`).</truth>
    <truth>/list fields per row: Name, Cron, Timezone, Next (RFC3339).</truth>
    <truth>Both outputs are deterministic (sorted by slug) so tests are stable.</truth>
    <truth>Bot.SendPicker(text, options) builds an inline keyboard from PickerOption list via `tgbotapi.NewInlineKeyboardMarkup` and sends it.</truth>
    <truth>Dispatcher routes /pull → pull.Handle, /status → status.Render → send, /list → list.Render → send. Inline callback queries with Data prefix `start:` → pull.HandleCallback.</truth>
  </truths>
</must_haves>

  <action>
    1. Add to `internal/bot/bot.go`:
       - `func (b *Bot) SendPicker(text string, options []PickerOption) error` — note: import the option type without creating a cycle. To avoid bot→commands import (would create cycle), define a parallel type here: `type PickerOption struct { Label, CallbackData string }`. Have `commands.PickerOption` be a type alias to `bot.PickerOption`. Update commands package accordingly. Then SendPicker builds `tgbotapi.NewInlineKeyboardMarkup(rows...)` with one button per option, attaches to message, sends.
       - Or simpler: leave PickerSender as an interface in commands package, and have bot.Bot implement a method `SendPicker(text string, options []commands.PickerOption)` via a shim in main.go (a small adapter struct). I'll go with the shim to keep the import direction clean. So `bot/bot.go` learns a generic `SendKeyboard(text string, buttons []struct{ Label, Callback string }) error` and the adapter wraps that.

    2. Update `internal/handler/dispatcher.go`:
       - Field additions: `Pull *commands.Pull`, `StatusCmd *commands.Status`, `ListCmd *commands.List`. Setter or constructor with all three.
       - In handleCommand: route `pull` → Pull.Handle(sender); `status` → sender.Send(StatusCmd.Render()); `list` → sender.Send(ListCmd.Render()).
       - In Handle: if `update.CallbackQuery != nil`, route to Pull.HandleCallback(sender, update.CallbackQuery.Data). Acknowledge via `bot.API.Request(tgbotapi.NewCallback(callbackID, ""))` to remove the loading spinner — add a `CallbackAcker` interface on the sender, or just include this call directly if dispatcher gets a reference to the API. Simpler: include an `AckCallback(callbackID string)` method on bot.Sender (interface widened); recordingSender implements as no-op.

    3. Update bot.IsAuthorised already handles callback queries via update.FromChat() — verify and leave.

    4. Wire in main.go: build CronBus, Pull, Status, List, dispatcher with all three, plus the adapter that lets bot.Bot satisfy commands.PickerSender. Start CronBus.Run(ctx) goroutine. Scheduler handler now calls `bus.Fire(slug, time.Now())`.
  </action>
